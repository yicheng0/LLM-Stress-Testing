# LLM API 压测脚本增强实施计划

## 1. 项目背景

### 1.1 当前问题

现有压测脚本 `llm_load_test.py` 存在以下关键缺陷：

| 问题类型 | 具体表现 | 影响 |
|---------|---------|------|
| **TTFT 缺失** | `RequestResult.ttft_sec` 字段始终为 `None` | 无法评估 Prefill 性能，无法区分排队/推理阶段 |
| **TPS 缺失** | 只有 TPM，没有 TPS（Tokens Per Second） | 无法评估实时吞吐能力 |
| **非流式模式** | 硬编码 `stream=False` | 无法准确测量首 token 到达时间 |
| **统计维度不足** | 缺少 TTFT、Decode 时间的分布统计 | 无法定位性能瓶颈 |
| **数据偏差** | 测试结果与实际表现不符 | 影响容量规划和性能优化决策 |

### 1.2 客户需求

根据客户提供的测试数据，需要支持：

**批次测试场景**：
```
批次1: 并发120, 输入系数X4  → RPM 47.7, Total-TPM 22.1W
批次2: 并发240, 输入系数X4  → RPM 92.7, Total-TPM 43.3W
批次3: 并发240, 输入系数X8  → RPM 85.7, Total-TPM 53.4W
```

**TTFT 矩阵测试**：
```
200 tokens   → TTFT 1340ms
2k tokens    → TTFT 2482ms
20k tokens   → TTFT 4361ms
120k tokens  → TTFT 3368ms
```

**必需指标**：
- TPM（Input/Output/Total）
- RPM（Requests Per Minute）
- TPS（Tokens Per Second）
- TTFT（Time To First Token）：average, P50, P95, P99
- 延迟（Latency）：average, P50, P95, P99
- 成功率、错误分布

### 1.3 实施目标

1. **准确测量 TTFT**：通过流式响应捕获首 token 到达时间
2. **完善指标体系**：添加 TPS、TTFT 分布、Decode 时间统计
3. **支持测试矩阵**：支持多输入规模 × 多并发级别的批量测试
4. **提升数据准确性**：优化时间测量、token 计数、统计方法
5. **增强报告质量**：生成更详细的分析报告和可视化数据

```python
def _parse_stream_usage(self, stream_text: str) -> Tuple[int, int]:
    """
    解析 SSE 流中的 usage 信息
    
    OpenAI 格式示例：
    data: {"choices":[{"delta":{"content":"Hello"}}]}
    data: {"choices":[{"delta":{}}],"usage":{"completion_tokens":10,"total_tokens":110}}
    data: [DONE]
    
    返回: (output_tokens, total_tokens)
    """
    output_tokens = 0
    total_tokens = 0
    
    for line in stream_text.split('\n'):
        line = line.strip()
        if line.startswith('data: '):
            data_str = line[6:]  # 去掉 "data: " 前缀
            if data_str == '[DONE]':
                break
            try:
                data = json.loads(data_str)
                if 'usage' in data:
                    usage = data['usage']
                    output_tokens = int(usage.get('completion_tokens') or usage.get('output_tokens') or 0)
                    total_tokens = int(usage.get('total_tokens') or 0)
            except json.JSONDecodeError:
                continue
    
    return output_tokens, total_tokens
```

## 5. 风险与缓解措施

### 5.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **SSE 格式不兼容** | TTFT 测量失败 | 中 | 1. 先测试小规模流式请求<br>2. 添加详细日志记录 SSE 格式<br>3. 支持多种 SSE 格式解析 |
| **流式处理性能问题** | 高并发下客户端 CPU 过高 | 低 | 1. 使用 `iter_any()` 而非 `iter_chunked(1)`<br>2. 异步处理避免阻塞<br>3. 性能测试验证 |
| **Token 计数偏差** | TPM 统计不准确 | 中 | 1. 优先使用 API 返回的 usage<br>2. 添加偏差统计和警告<br>3. 文档说明估算方法 |
| **矩阵测试时间过长** | 测试效率低 | 低 | 1. 支持并行测试（未来优化）<br>2. 可配置每个测试点时长<br>3. 支持断点续测 |

### 9.4 常见问题

**Q1：为什么 TTFT 显示为 N/A？**
A：需要使用 `--enable-stream` 参数启用流式模式。

**Q2：如何解释 TTFT > 总延迟的情况？**
A：这是异常情况，可能是时间测量错误或网络问题，需要检查详细日志。

**Q3：矩阵测试需要多长时间？**
A：时间 = 输入规模数 × 并发级别数 × 每个测试点时长 + 冷却时间。例如：3 × 3 × 60s + 9 × 10s = 630s ≈ 10.5 分钟。

**Q4：如何选择合适的并发数？**
A：建议从低并发开始（如 50），逐步增加，观察成功率和延迟变化，找到拐点。

**Q5：测试结果与客户数据偏差大怎么办？**
A：检查以下因素：
1. 网络环境是否一致
2. 测试时段是否一致（避开高峰期）
3. API 配置是否一致（如限流策略）
4. 客户端性能是否充足
